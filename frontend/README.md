# VidGram Frontend – Master Documentation

## Project Overview

**What the project does:**
The VidGram frontend is a web-based dashboard that allows marketing teams and campaign managers to interact with the influencer risk assessment engine. It visualizes authenticity scores, highlights fraud risks, monitors campaign budgets, and displays actionable alerts generated by the backend systems.

**Problem being solved:**
While the backend risk engines compute complex fraud probabilities, non-technical marketing teams need an intuitive, visual way to understand these metrics. This frontend transforms raw data arrays and risk classifications into easy-to-read charts, clear warnings, and manageable campaign timelines, preventing budget waste effectively.

**High-level workflow:**
User inputs influencer data → Backend API computes risk → Frontend receives ScoringResult → Renders specific views (Analyze, Budget, Monitor) based on risk classifications.

### File Purposes
- **index.html**: The unified home screen that summarizes active campaigns and high-level risk metrics.
- **analyze.html**: Consumes the `ScoringResult` from the backend to show the 0-100 score, evidence list, and reach calculations.
- **budget.html**: Focuses purely on financial impact, displaying the estimated loss and genuine reach values.

## Data Flow

### End-to-End Example: Rendering an Influencer Risk Profile

**Step 1: User Request**
User searches for "dataset_creator" in the UI.

**Step 2: API Call**
Frontend sends request to the Backend APIs.

**Step 3: Response Parsing**
Frontend receives the normalized `ScoringResult`:
```json
{
  "authenticity_score": 32,
  "risk_level": "High",
  "campaign_risk": {
    "estimated_loss": 14960.0
  }
}
```

**Step 4: DOM Updates**
- `analyze.html` renders a red "High Risk" badge and a 32/100 dial chart.
- `alerts.html` adds a new high-severity notification card.
- `budget.html` updates the "Potential Waste" graph with $14,960.00.

## Input/Output Contract

### Consumed Schema
The frontend strictly relies on the following schema shapes provided by the backend:

```json
{
    "authenticity_score": "Number (0-100)",
    "risk_level": "String (Low/Medium/High)",
    "evidence": "Array of Strings",
    "alerts": "Array of Objects (type, severity, message)"
}
```

### Error Handling
If the backend returns a 500 error or validation failure, the frontend defaults to a "Service Unavailable" state, preventing the user from approving campaigns without a valid risk score.
